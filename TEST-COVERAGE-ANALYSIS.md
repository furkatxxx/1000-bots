## Рекомендации по приоритетам тестирования

### Приоритет 1 — Утилиты и чистые функции (быстрая победа)

**Файл: `src/lib/utils.ts`** (108 строк)

Чистые функции без побочных эффектов — идеальные кандидаты для первых тестов:

- `formatDate()` — форматирование дат на русском, обработка невалидных дат
- `formatDateShort()` — краткий формат даты
- `getDifficultyLabel()` / `getDifficultyColor()` — маппинг значений
- `getCompetitionLabel()` / `getCostLabel()` — маппинг значений
- `concurrentMap()` — параллельное выполнение с ограничением (граничные случаи: пустой массив, ошибка в одном из промисов, ограничение конкурентности)

**Ожидаемые тесты:** ~15-20 тест-кейсов

### Приоритет 2 — Фильтрация трендов (критичная бизнес-логика)

**Файл: `src/lib/ai-brain.ts`** — функция `filterTrends()`

Эта функция определяет, какие тренды попадают в генерацию, а какие отсеиваются. Ошибка здесь = мусорные идеи или потеря хороших трендов.

Что тестировать:
- Спортивные тренды отфильтровываются (`NBA finals`, `матч Динамо`)
- Развлекательные тренды отфильтровываются (`Netflix`, `Тейлор Свифт`)
- Бизнес-тренды ПРОХОДЯТ фильтр (`AI copilot for sales`, `автоматизация складов`)
- Фильтрация по категории (`sports`, `entertainment`, `gaming`)
- Граничные случаи: пустой массив, тренд без категории, регистронезависимость

**Ожидаемые тесты:** ~20 тест-кейсов

### Приоритет 3 — Проверка здоровья источников

**Файл: `src/lib/health-check.ts`** (223 строки)

Функции `isHealthyEnough()` и `sendHealthTelegramAlert()` содержат важную логику принятия решений:

- `isHealthyEnough()` — порог 60%: при 2 из 4 работающих (50%) → отмена генерации
- Граничные случаи: 0 источников, все работают, все сломаны, ровно на границе 60%
- Формирование сообщений для Telegram (корректность HTML-разметки)
- `checkSource()` — обработка ошибок, замер времени

**Ожидаемые тесты:** ~10-15 тест-кейсов

### Приоритет 4 — Пресеты фокуса генерации

**Файл: `src/lib/focus-presets.ts`** (322 строки)

- `buildFocusBlock()` — корректное построение промпт-блока для 1, 2 или 3 пресетов
- Валидация: невалидный `PresetId`, пустой массив, дублирование пресетов
- Формат выходных данных (наличие обязательных полей в каждом пресете)

**Ожидаемые тесты:** ~10 тест-кейсов

### Приоритет 5 — Семантическая дедупликация

**Файл: `src/lib/ai-brain.ts`** — функция `semanticDedup()`

- Поведение при ≤3 идеях (возвращает без вызова API)
- Корректный парсинг ответа AI (номера дубликатов)
- Обработка ошибок API (таймаут, невалидный ключ)
- Граничные случаи: все идеи уникальны, все дубликаты

**Ожидаемые тесты:** ~8-10 тест-кейсов (с моками Anthropic API)

### Приоритет 6 — API-роуты

**Папка: `src/app/api/`** (24 роута)

Наиболее критичные роуты для тестирования:
- `POST /api/reports` — создание отчёта с проверкой здоровья
- `GET /api/reports/[id]` — получение отчёта
- `POST /api/ideas` — создание идеи с валидацией
- `POST /api/pipeline/stage-1` — первый этап генерации
- `POST /api/telegram/send-top` — отправка топа в Telegram

Нужны моки для: Prisma, Anthropic API, внешних API (Google Trends, Reddit и т.д.)

**Ожидаемые тесты:** ~30-40 тест-кейсов

## Рекомендуемый стек для тестирования

```
vitest              — быстрый, нативная поддержка TypeScript и ESM
@testing-library/react — тестирование React-компонентов
msw                 — моки HTTP-запросов (внешние API)
```

### Почему Vitest, а не Jest:
- Нативная поддержка TypeScript без конфигурации
- Совместим с Vite/Next.js (ESM из коробки)
- Быстрее Jest на проектах с TypeScript
- API совместим с Jest (легко начать)
